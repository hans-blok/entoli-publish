# Entoli Execution & Agent Model — Relations and Cardinalities This view defines the **canonical relationships and cardinalities** between core concepts in the Entoli ecosystem. It supports **traceability, determinism, and contract-driven execution**. --- ## 1. Value Stream Structure - **Value Stream → Value Stream Phase** - *is decomposed into* - `1 → 1..n` - Each value stream consists of one or more phases - Each phase belongs to exactly one value stream --- ## 2. Agent Positioning - **Agent → Value Stream Phase** - *operates in* - `n → 1` - Each agent operates in one primary phase - **Agent → Formation Phase** - *is positioned in* - `n → 1` - Formation phase is a classification attribute - **Agent → Operation Mode** - *operates under* - `n → 1` - Defines how the agent operates (stable characteristic) --- ## 3. Agent and Capabilities - **Agent → Agent Contract** - *exposes capabilities via* - `1 → 1..n` - An agent exposes one or more capabilities - Each contract belongs to exactly one agent --- ## 4. Capability Definition (Agent Contract) - **Agent Contract → Intervention Type** - *defines* - `n → 1` - Each capability has exactly one intervention type - **Agent Contract → Source Regime** - *applies* - `n → 1` - Defines how the capability uses sources - **Agent Contract → Agent Contract Rule** - *is constrained by* - `1 → 0..n` - A contract may have zero or more rules --- ## 5. Input and Output Definition - **Agent Contract → Artifact Type (input)** - *requires* - `1 → 0..n` - Input may be optional - **Agent Contract → Artifact Type (output)** - *produces* - `1 → 1..n` - Each capability must produce at least one artifact --- ## 6. Execution Layer - **Execution → Agent Contract** - *executes* - `n → 1` - Each execution is an instance of one contract - **Orchestration Run → Execution** - *consists of* - `1 → 1..n` - A run groups one or more executions --- ## 7. LLM Interaction - **Execution → LLM Call** - *invokes* - `1 → 0..n` - An execution may result in multiple LLM calls --- ## 8. Artifact Flow - **Execution → Artifact (produces)** - *produces* - `1 → 1..n` - Each execution produces one or more artifacts - **Execution ↔ Artifact (consumes)** - *consumes* - `n ↔ n` - Executions may consume multiple artifacts - Artifacts may be reused across executions --- ## 9. Artifact Typing - **Artifact → Artifact Type** - *is of type* - `n → 1` - Each artifact has exactly one type ## 10. Doctrine Binding - **Agent Contract Rule → Doctrine Rule** - *derives from* - `n → 1` - Each agent contract rule is a contextual binding of exactly one doctrine rule - A doctrine rule may be bound to zero or many agent contracts via agent contract rules --- ## 11. Key Design Constraints - **Intervention Type is defined at capability level (Agent Contract), not Agent level** - **Source Regime is defined per capability** - **Execution is strictly bound to one contract** - **Produced and consumed artifacts are distinct relations** - **Determinism is enforced through one-to-many and many-to-one constraints (avoid uncontrolled many-to-many)** --- ## 11. Architectural Principle > **Agents provide capabilities, contracts define behavior, coordination prepares execution, and execution produces traceable artifacts.** Agent Contract Rules are contextual bindings that derive from canonical Doctrine Rules and constrain the behavior of specific capabilities.
